# ADR-003: Direct DbContext, No Repository Wrapper over EF Core

## Status
Accepted

## Context
A common .NET pattern adds a repository layer (`IUserRepository`, `IOrderRepository`) on top of EF Core's `DbContext`. The stated goal is testability and abstraction over the data store.

In practice, EF Core itself implements the Repository pattern (`DbSet<T>`) and the Unit of Work pattern (`DbContext`). Adding another layer:
- Requires mocking the mock (a fake of a fake of the database)
- Prevents LINQ composition across entity sets (`_db.Orders.Include(o => o.User)`)
- Breaks global query filters (tenant isolation, soft-delete) that are applied at the `DbContext` level
- Adds 200–400 lines of boilerplate per entity with zero functional addition

## Decision

Application services and handlers **inject `AppDbContext` directly** (or an `IAppDbContext` interface for seam-testability).

```csharp
public class GetOrderHandler(AppDbContext db) : IRequestHandler<GetOrderQuery, OrderDto>
{
    public async Task<OrderDto> Handle(GetOrderQuery query, CancellationToken ct)
        => await db.Orders
            .Where(o => o.Id == query.OrderId)
            .Select(o => new OrderDto(o.Id, o.Status, o.TotalAmount))
            .FirstOrDefaultAsync(ct)
            ?? throw new NotFoundException(nameof(Order), query.OrderId);
}
```

For integration testing, use `Testcontainers` with a real SQL Server instance. For unit testing, use EF Core's in-memory provider only for read-path logic where a real DB would be overkill.

## Consequences

**Easier**: Full LINQ expressiveness, global query filters apply automatically, no abstraction layer to maintain, integration tests with Testcontainers are fast and accurate.

**Harder**: Handlers directly depend on `AppDbContext`. If switching ORMs, handlers change. This is an acceptable risk — switching ORMs in a production system is a major undertaking regardless of repository abstraction, and the indirection does not meaningfully reduce the blast radius.

**Tradeoff accepted**: ORM flexibility via repository abstraction is a false promise for most projects. Real testability comes from Testcontainers, not from mocking.

## Alternatives Considered

- **Generic repository `IRepository<T>`**: rejected — EF Core's `DbSet<T>` IS the generic repository
- **Per-aggregate repository interfaces**: partially useful for domain modeling but still rejected as an application-layer pattern — adds interfaces without addind isolation
