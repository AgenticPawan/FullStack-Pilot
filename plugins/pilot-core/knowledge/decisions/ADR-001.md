# ADR-001: Permissions-Only Authorization (No Role Checks in Application Code)

## Status
Accepted

## Context
Authorization models fall into two broad categories: **role-based** (is the user an Admin?) and **permission/claim-based** (does the user have the `orders:write` claim?). Many teams start with roles because they map naturally to user categories, then discover that roles expand — Admin, SuperAdmin, ReadOnlyAdmin, TenantAdmin — until every new feature needs a new role.

FullStack-Pilot's governance stack sees this play out consistently: role proliferation creates fragile conditional logic scattered across endpoints and components; adding a new capability requires changing the authorization model, not just the code.

## Decision

All application-code authorization checks use **fine-grained permissions/claims**, never role names directly.

- **Angular route guards**: check `user.permissions.includes('orders:read')`, not `user.roles.includes('Admin')`
- **.NET policy definitions**: `policy.RequireClaim("permission", "orders:write")`, not `[Authorize(Roles = "Admin")]`
- **Role-to-permission mapping** happens at the IdP (identity provider) or at token issuance, not in application code

## Consequences

**Easier**: Adding a new capability is a permission addition — no controller attribute or guard change needed. Cross-role capabilities (a Supervisor who can approve but not delete) are expressed as claim sets, not role combinations.

**Harder**: Initial setup requires a permission catalog defined upfront. Teams new to claim-based auth need onboarding.

**Tradeoff accepted**: The IdP becomes the source of truth for role-to-permission mapping. This is the correct coupling — authorization policy belongs with identity, not scattered in every service.

## Alternatives Considered

- **Role checks inline**: `[Authorize(Roles = "Admin")]` — rejected: role proliferation, hard to audit, tight coupling between feature code and user categories
- **Policy per feature**: `[Authorize(Policy = "CanEditOrders")]` with role checks inside the policy — partially better but still ties policy to roles at the infrastructure level
