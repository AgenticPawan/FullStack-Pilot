# ADR-004: `takeUntilDestroyed` over `ngOnDestroy` + Subject

## Status
Accepted

## Context
Angular components that subscribe to observables must clean up those subscriptions when the component is destroyed. The historical pattern uses `ngOnDestroy` and a `Subject`:

```typescript
private destroy$ = new Subject<void>();

ngOnInit() {
  this.service.getData().pipe(takeUntil(this.destroy$)).subscribe(...);
}
ngOnDestroy() {
  this.destroy$.next();
  this.destroy$.complete();
}
```

This pattern:
- Requires 5+ lines of boilerplate per component
- Is easy to forget â€” if `ngOnDestroy` is not declared, subscriptions leak silently
- Compounds when multiple component lifecycle hooks are involved
- Conflicts with future Zoneless Angular (where `ngOnDestroy` hook timing may change)

Angular 16 introduced `DestroyRef` and `takeUntilDestroyed` as the platform-sanctioned replacement.

## Decision

All Angular components and directives use `takeUntilDestroyed(this.destroyRef)` for subscription cleanup. `ngOnDestroy` + Subject is no longer written for new components.

```typescript
import { Component, inject, DestroyRef } from '@angular/core';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';

@Component({ ... })
export class OrderListComponent {
  private destroyRef = inject(DestroyRef);

  ngOnInit() {
    this.orderService.orders$
      .pipe(takeUntilDestroyed(this.destroyRef))
      .subscribe(orders => this.orders = orders);
  }
}
```

For constructor-initialized subscriptions, `this.destroyRef` can be omitted (the current injection context is used automatically):
```typescript
constructor() {
  this.service.stream$.pipe(takeUntilDestroyed()).subscribe(...);
}
```

## Consequences

**Easier**: Subscription cleanup is one method call, not a lifecycle hook + Subject + two method bodies. Compatible with standalone components. Framework manages cleanup timing correctly with future Zoneless support.

**Harder**: Requires Angular 16+. Projects on Angular 15 or below must continue with the Subject pattern until they upgrade. The `angular-memory-leaks` skill enforces this check per version.

**Tradeoff accepted**: Angular 15 projects are on a supported upgrade path. All net-new greenfield work targets Angular 17+.

## Alternatives Considered

- **`async` pipe in templates**: preferred where possible â€” no manual cleanup at all â€” but does not cover all subscription patterns (e.g., subscriptions that trigger side effects)
- **`takeUntil(this.destroy$)` pattern**: rejected for new code â€” deprecated by the framework's own guidance
