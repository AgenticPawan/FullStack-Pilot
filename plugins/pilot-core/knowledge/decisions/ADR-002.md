# ADR-002: GUID Entity Keys (Not int/identity)

## Status
Accepted

## Context
The two most common primary key strategies for relational entities in .NET / EF Core are:

- **`int IDENTITY`** (auto-increment): database generates sequential integers
- **`Guid` (UUID v4 or v7)**: application generates a random or ordered UUID

Multi-tenant SaaS applications on SQL Server, and APIs that expose resource IDs in URLs, interact with both strategies differently.

## Decision

All domain entities use **`Guid` primary keys** generated by the application (not the database).

EF Core configuration:
```csharp
modelBuilder.Entity<Order>().Property(o => o.Id)
    .HasDefaultValueSql("NEWSEQUENTIALID()");  // SQL Server: ordered GUIDs reduce fragmentation
```
Or prefer `Guid.CreateVersion7()` in .NET 9+ for time-ordered GUIDs without a SQL default.

## Consequences

**Easier**: IDs can be generated in the application layer before the INSERT — enables optimistic pre-allocation, deterministic integration tests (seed a known GUID), and safe ID exposure in URLs without leaking row count. Cross-database merges (multi-region, import/export) have no collision risk.

**Harder**: GUIDs are larger (16 bytes vs 4), less human-readable in logs, and random UUIDs fragment the clustered index. Mitigated by `NEWSEQUENTIALID()` or `Guid.CreateVersion7()`.

**Tradeoff accepted**: Index fragmentation from random GUIDs is solved by ordered GUIDs, not by reverting to integers. The benefits in a multi-tenant, distributed-team context outweigh the storage overhead.

## Alternatives Considered

- **`int IDENTITY`**: rejected — leaks row count via API, forces a database roundtrip before knowing the ID, collision risk in multi-master scenarios
- **HiLo sequences**: rejected — adds configuration complexity for marginal read performance gain that does not matter at typical application scales
