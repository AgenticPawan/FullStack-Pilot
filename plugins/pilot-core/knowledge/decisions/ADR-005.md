# ADR-005: Tenant Filter at DbContext Level, Not Controller Level

## Status
Accepted

## Context
Multi-tenant SaaS applications need to ensure that every data query is scoped to the current tenant. There are two common enforcement points:

1. **Controller/handler level**: each handler adds `Where(x => x.TenantId == tenantId)` to every query
2. **DbContext level**: a global query filter applies `Where(x => x.TenantId == tenantId)` automatically to every entity that has a `TenantId`

Option 1 is fragile. Developers forget the filter. New endpoints skip it under deadline pressure. Code review catches some instances but not all. The result: occasional data leaks between tenants that are hard to reproduce and costly to remediate.

## Decision

Tenant isolation is enforced via **EF Core global query filters** applied in `OnModelCreating`, scoped to the current `ITenantContext` service:

```csharp
public class AppDbContext(DbContextOptions options, ITenantContext tenantContext)
    : DbContext(options)
{
    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        // All entities implementing ITenantScoped get the filter automatically
        foreach (var entityType in modelBuilder.Model.GetEntityTypes()
            .Where(e => typeof(ITenantScoped).IsAssignableFrom(e.ClrType)))
        {
            modelBuilder.Entity(entityType.ClrType)
                .HasQueryFilter(BuildTenantFilter(entityType.ClrType, tenantContext));
        }
    }
}
```

Bypassing the filter for cross-tenant admin operations requires an explicit `.IgnoreQueryFilters()` call that is visible in code review and flagged by the `sql-multitenancy` skill.

## Consequences

**Easier**: No developer can accidentally leak tenant data by forgetting a WHERE clause. New entities automatically inherit isolation by implementing `ITenantScoped`. The filter appears in exactly one place — EF Core enforces it at the SQL generation layer.

**Harder**: `IgnoreQueryFilters()` must be used deliberately for admin queries or migration scripts. These are easier to audit because they are explicit. Seed data and migrations that run before a tenant context exists require a `NullTenantContext` implementation.

**Tradeoff accepted**: The additional setup for `NullTenantContext` in migrations is one-time work; the security guarantee is continuous.

## Alternatives Considered

- **RLS (Row-Level Security) in SQL Server**: provides a database-layer guarantee but duplicates logic (filter in both DbContext and the DB) and complicates migrations. Considered for highly regulated environments where defense-in-depth is mandatory — see `sql-multitenancy` skill.
- **Per-handler WHERE clauses**: rejected as the primary enforcement strategy — too easy to forget, no automated enforcement without custom analyzers
